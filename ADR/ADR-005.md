# ADR-005 — Por qué Docker para el backend

- **Estado:** Aceptado
- **Fecha:** 2026-06-30
- **Decisores:** Arquitectura inicial del proyecto

---

## Contexto

El backend de FiscalOS (FastAPI) necesita ser empaquetado y desplegado en Render. Las opciones van desde buildpacks gestionados por la plataforma hasta contenedores Docker que el equipo controla completamente.

La decisión de cómo empaquetar el backend tiene implicaciones en la reproducibilidad del ambiente, la velocidad de deploy, la paridad entre desarrollo y producción, y la portabilidad futura del sistema.

---

## Problema

Necesitamos una estrategia de empaquetado para FastAPI que:
1. Garantice paridad exacta entre el ambiente de desarrollo local y producción
2. Sea deployable en Render sin complejidad de infraestructura adicional
3. Permita definir dependencias del sistema (no solo de Python)
4. Facilite el onboarding de nuevos ingenieros al proyecto
5. Sea portable si en el futuro se cambia de proveedor de hosting

---

## Alternativas evaluadas

### Buildpacks de Render (sin Docker)
**Ventajas:** Cero configuración. Render detecta el proyecto Python automáticamente. No hay que mantener un Dockerfile.
**Desventajas:**
- El ambiente de producción es opaco — no se puede reproducir localmente con exactitud
- Las versiones de Python y dependencias del sistema son controladas por Render, no por el equipo
- Si Render cambia el buildpack, el comportamiento de producción puede cambiar sin aviso
- Dificulta agregar dependencias del sistema operativo (herramientas de procesamiento de PDF, por ejemplo)

### Serverless Functions (AWS Lambda / Vercel Functions)
**Ventajas:** Costo reducido a escala baja, sin gestión de servidores.
**Desventajas:**
- FastAPI no está optimizado para cold starts de serverless — el tiempo de inicio puede ser alto
- Los workers programados (APScheduler) no funcionan en serverless — necesitarían un servicio separado (EventBridge, cron jobs)
- Los límites de tiempo de ejecución de serverless son restrictivos para operaciones que pueden demorar
- El modelo de programación async de FastAPI se diseñó para procesos long-running, no para funciones efímeras

### Heroku (con buildpacks)
**Ventajas:** Simplicidad, herramienta conocida.
**Desventajas:** Heroku dejó de ser una opción competitiva en costo/performance. Render es el sucesor espiritual con mejor DX y pricing.

### Docker
**Ventajas:**
- El `Dockerfile` es la especificación exacta del ambiente — no hay diferencia entre local y producción
- `docker-compose.yml` permite levantar FastAPI + Supabase local en un solo comando
- Las dependencias del sistema operativo son parte del Dockerfile — sin sorpresas
- Render tiene soporte nativo de Docker — el Dockerfile es el único artefacto de deployment
- Si se cambia de Render a otro proveedor (AWS ECS, GCP Cloud Run, etc.), el Docker image funciona igual
- Los ingenieros nuevos clonan el repo, corren `docker-compose up` y tienen el ambiente completo

**Desventajas:**
- Requiere escribir y mantener un Dockerfile
- El tiempo de build del Docker image es ligeramente mayor que buildpacks simples
- Los desarrolladores necesitan Docker Desktop instalado (o una alternativa como OrbStack)

---

## Decisión

**Docker** es la estrategia de empaquetado para FastAPI.

La razón principal es la reproducibilidad: el ambiente de producción debe ser idéntico al ambiente de desarrollo. Con buildpacks, hay una brecha entre lo que el desarrollador ve localmente y lo que corre en producción — y esa brecha es el origen de errores difíciles de depurar. Con Docker, el Dockerfile es el contrato y no hay lugar a divergencias.

La portabilidad es el segundo factor: la decisión de usar Render es pragmática, no permanente. Docker garantiza que si en algún momento el proveedor de hosting cambia, no hay reescritura de infraestructura.

---

## Consecuencias

### Positivas
- Paridad total entre development, staging y production
- `docker-compose up` levanta el stack completo localmente (FastAPI + Supabase)
- El Dockerfile documenta implícitamente las dependencias del sistema
- Portabilidad a cualquier proveedor que soporte Docker (que es prácticamente todos)
- Las builds son reproducibles y verificables

### Negativas
- Los desarrolladores necesitan Docker instalado — pequeña fricción de setup inicial
- Las builds de Docker tienen un overhead de tiempo vs. recargas en vivo de Python
- El Dockerfile debe mantenerse actualizado cuando se agregan dependencias del sistema

### Especificaciones del Dockerfile
- Imagen base: `python:3.12-slim` — sin dependencias innecesarias del sistema operativo
- Multi-stage build: etapa de `builder` para instalar dependencias, etapa `runner` para ejecutar
- El usuario de runtime no es `root` — buena práctica de seguridad en contenedores
- Las dependencias de Python se instalan desde `requirements.txt` — no desde pyproject.toml para simplicidad

### Decisiones derivadas
- `docker-compose.yml` incluye: FastAPI, Supabase local (mediante `supabase start`), variables de entorno de desarrollo
- Las variables de entorno en producción se inyectan vía el dashboard de Render, nunca en el Docker image
- El `.dockerignore` excluye: `.env`, `__pycache__`, `.git`, archivos de test
- El health check del contenedor apunta a `GET /health` en FastAPI — Render lo usa para verificar que el servicio está listo
