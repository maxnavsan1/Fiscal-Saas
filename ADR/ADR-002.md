# ADR-002 — Por qué PostgreSQL como base de datos

- **Estado:** Aceptado
- **Fecha:** 2026-06-30
- **Decisores:** Arquitectura inicial del proyecto

---

## Contexto

FiscalOS necesita persistir datos estructurados con relaciones claras entre entidades: Organization → Client → Document, Filing, DocumentRequest. Los datos fiscales son por naturaleza relacionales: una declaración tiene un tipo, un período, un cliente, etapas con timestamps, y documentos adjuntos. La integridad referencial no es opcional.

Adicionalmente, el modelo de seguridad del producto depende de Row Level Security (RLS) — la capacidad de la base de datos de aplicar políticas de acceso a nivel de fila, según el usuario autenticado. Esta es una decisión de seguridad crítica: si la base de datos no soporta RLS nativo, la seguridad multi-tenant depende enteramente del código de la aplicación, lo cual es un riesgo inaceptable.

---

## Problema

Necesitamos una base de datos que:
1. Soporte Row Level Security nativo a nivel de motor
2. Modele datos relacionales con integridad referencial (foreign keys, constraints)
3. Soporte JSONB para metadatos flexibles (activity log metadata, filing metadata)
4. Tenga un ecosistema maduro con herramientas de migración probadas
5. Sea compatible con Supabase como plataforma managed
6. Tenga un historial probado en aplicaciones B2B de escala

---

## Alternativas evaluadas

### MySQL / MariaDB
**Ventajas:** Muy conocida, rápida para reads simples, amplia adopción.
**Desventajas:** Row Level Security en MySQL no existe de forma nativa comparable a PostgreSQL. Habría que implementar el aislamiento multi-tenant completamente en código de aplicación. JSONB es menos capaz. No es compatible con Supabase. Historial de tipos de datos menos estricto.

### MongoDB (NoSQL)
**Ventajas:** Esquema flexible, bueno para documentos no estructurados.
**Desventajas:** Las relaciones entre entidades son complejas de mantener. No hay RLS. Los datos fiscales son inherentemente relacionales y estructurados — MongoDB complica la integridad de datos. Los joins son costosos. No es compatible con Supabase.

### CockroachDB
**Ventajas:** Distribuida, SQL compatible, alta disponibilidad.
**Desventajas:** Sobre-ingeniería para el MVP. El modelo de pricing es más costoso. La compatibilidad con Supabase no existe. Para el volumen esperado en los primeros 2-3 años, PostgreSQL en Supabase es suficiente.

### PlanetScale (MySQL serverless)
**Ventajas:** Serverless, branching de base de datos, buena DX.
**Desventajas:** Basado en MySQL (sin RLS nativo). No soporta foreign keys (por Vitess). Para un sistema que requiere integridad referencial estricta, esta limitación es inaceptable.

### SQLite
**Ventajas:** Cero configuración, ideal para desarrollo.
**Desventajas:** No es una base de datos de producción para una aplicación multi-tenant con concurrencia real. Sin RLS. Sin replicación.

### PostgreSQL
**Ventajas:**
- Row Level Security nativo y robusto — la columna vertebral del modelo de seguridad
- JSONB para metadatos flexibles con índices eficientes
- Tipos de datos ricos: UUID, JSONB, arrays, enums, timestamps con timezone
- Integridad referencial completa: foreign keys, constraints, check constraints
- Ecosistema maduro: Alembic para migraciones, SQLAlchemy para ORM, pg_dump para backups
- Compatible con Supabase como plataforma managed
- Realtime basado en logical replication de Postgres
- Probado en producción a escala por décadas
- Open source, sin lock-in de vendor

**Desventajas:**
- Más complejo de operar que bases de datos managed más simples (mitigado por Supabase)
- Las migraciones requieren cuidado con datos en producción

---

## Decisión

**PostgreSQL** vía **Supabase** es la base de datos del sistema.

PostgreSQL es la única base de datos relacional con soporte de RLS lo suficientemente robusto para soportar el modelo de seguridad multi-tenant de FiscalOS. Esta sola razón es determinante: si el modelo de seguridad dependiera del código de aplicación, cualquier bug de autorización comprometería datos de clientes. Con RLS en la base de datos, el error de aplicación retorna filas vacías, no datos incorrectos.

---

## Consecuencias

### Positivas
- RLS como capa de seguridad de datos — independiente del código de aplicación
- JSONB para el `metadata` del activity log y metadatos de filings, sin esquemas separados
- Supabase Realtime funciona sobre logical replication de Postgres — sin infraestructura adicional
- Alembic maneja las migraciones de forma explícita y versionada
- Los tipos generados por Supabase CLI (`supabase gen types`) mantienen los tipos de TypeScript sincronizados con el esquema

### Negativas
- Las migraciones en producción requieren cuidado: columnas NOT NULL en tablas con datos requieren defaults o migración en dos fases
- El RLS bien diseñado requiere conocimiento de Postgres — no cualquier desarrollador puede escribir políticas correctamente sin revisión

### Decisiones derivadas
- Todas las tablas del dominio tienen `organization_id` como primer filtro en las políticas de RLS
- Los UUIDs son el tipo de ID estándar — no IDs incrementales (evitan enumeración y facilitan la generación en el cliente)
- El `activity_log` usa políticas de solo INSERT para garantizar su inmutabilidad
- Las migraciones se escriben con Alembic (FastAPI) y se revisan como parte del PR antes de aplicarse a staging
