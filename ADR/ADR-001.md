# ADR-001 — Por qué FastAPI como backend

- **Estado:** Aceptado
- **Fecha:** 2026-06-30
- **Decisores:** Arquitectura inicial del proyecto

---

## Contexto

FiscalOS requiere una capa de backend para manejar operaciones de negocio con side effects: crear una declaración dispara notificaciones y escribe en el audit log; aprobar un documento cambia su estado, cierra la solicitud vinculada y notifica al cliente. Estas operaciones no pueden vivir únicamente en el cliente del browser de Supabase porque requieren privilegios elevados (service role) y deben ejecutarse de forma atómica.

Adicionalmente, el sistema necesitará workers programados para enviar recordatorios de fechas límite fiscales — una necesidad de proceso en background que no existe en el frontend.

La pregunta es: ¿qué tecnología usar para este backend?

---

## Problema

Necesitamos un framework backend que:
1. Maneje operaciones con side effects complejos de forma limpia
2. Soporte workers y tareas programadas
3. Sea rápido de desarrollar sin sacrificar estructura
4. Tenga tipado y validación nativa para las APIs
5. Pueda ejecutarse en Docker
6. Sea familiar para el equipo actual o fácil de onboardear

---

## Alternativas evaluadas

### Node.js con Express o Fastify
**Ventajas:** Mismo lenguaje que el frontend (TypeScript), ecosistema NPM compartido, un solo lenguaje en el stack.
**Desventajas:** Express no tiene estructura opinionada, lo que lleva a APIs inconsistentes en equipos. Fastify es mejor pero requiere más configuración manual. La validación y documentación de APIs requiere librerías adicionales (Joi, Zod, OpenAPI).

### Django (Python)
**Ventajas:** Muy maduro, ORM incluido, admin panel gratis, Django REST Framework es robusto.
**Desventajas:** Más pesado que lo necesario. El paradigma de Django está orientado a MVC tradicional, no a APIs async. Para un API sin templates, es exceso.

### Ruby on Rails
**Ventajas:** Velocidad de desarrollo en el pasado.
**Desventajas:** No es la fortaleza del equipo. El ecosistema Python es más relevante para el futuro del producto (ML/IA para documentos fiscales en v4). No agrega valor diferencial.

### Go (Gin o Echo)
**Ventajas:** Performance excepcional, binarios pequeños, fuertemente tipado.
**Desventajas:** Curva de aprendizaje más pronunciada. El boilerplate es considerablemente mayor. Para el volumen de operaciones del MVP, Go es over-engineered.

### FastAPI (Python)
**Ventajas:**
- Validación y serialización automática con Pydantic
- Generación automática de documentación OpenAPI (Swagger + Redoc)
- Soporte async nativo
- Tipado con Python type hints
- Simple para operaciones CRUD, robusto para operaciones complejas
- APScheduler para workers programados
- Ecosistema Python facilita integración futura con ML/IA
- Comunidad activa, adopción creciente en startups y empresas

**Desventajas:**
- Segundo lenguaje en el stack (Python + TypeScript)
- El equipo debe mantener dos entornos de desarrollo
- Los tipos de Python y TypeScript no se comparten automáticamente (requiere generación de clientes tipados)

---

## Decisión

**FastAPI** es el framework elegido para el backend.

La decisión se inclina por FastAPI sobre Node.js porque:
1. La validación automática con Pydantic elimina una clase completa de bugs de runtime
2. La documentación OpenAPI automática acelera la integración frontend-backend
3. El ecosistema Python es estratégicamente más valioso para las fases futuras del producto (extracción de documentos con IA, procesamiento de XMLs fiscales del SAT)
4. La curva de onboarding es menor que Go, y la estructura es más opinionada que Express

La desventaja del segundo lenguaje en el stack se mitiga manteniendo la responsabilidad de FastAPI claramente delimitada: solo operaciones con side effects, nunca CRUD simple.

---

## Consecuencias

### Positivas
- APIs autodocumentadas desde el día uno
- Validación robusta de requests y responses con Pydantic
- Workers programados (APScheduler) en el mismo proceso
- Preparados para integración futura con librerías de ML/IA en Python

### Negativas
- El equipo necesita mantener dos entornos de desarrollo (Node.js + Python)
- Los tipos de TypeScript deben generarse o mantenerse manualmente a partir de los schemas de Pydantic
- Dos lenguajes en el stack aumentan la superficie cognitiva para nuevos ingenieros

### Decisiones derivadas
- Los schemas de Pydantic son la fuente de verdad del contrato API
- El cliente TypeScript de FastAPI se genera automáticamente desde la spec OpenAPI
- FastAPI corre en Docker para garantizar el mismo ambiente entre development, staging y production
- El split de responsabilidades (Supabase para lecturas / FastAPI para escrituras con side effects) debe mantenerse disciplinadamente para no crear un proxy CRUD innecesario
