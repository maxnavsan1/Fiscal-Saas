# ADR-004 — Por qué Magic Links como método de autenticación

- **Estado:** Aceptado
- **Fecha:** 2026-06-30
- **Decisores:** Arquitectura inicial del proyecto

---

## Contexto

FiscalOS tiene dos tipos principales de usuarios desde la perspectiva de autenticación:

1. **Clientes del despacho** — Dueños de negocios, personas físicas con actividad empresarial. Su relación con el sistema es esporádica: entran cuando reciben una notificación. No tienen incentivo para recordar una contraseña.

2. **Staff del despacho** — Contadores y administrativos. Su uso es diario. La seguridad de su cuenta es crítica porque tienen acceso a datos de todos sus clientes.

El método de autenticación afecta la seguridad, la fricción de onboarding y la experiencia cotidiana de ambos tipos de usuario.

---

## Problema

Necesitamos un método de autenticación que:
1. Sea lo suficientemente seguro para proteger datos fiscales sensibles
2. Minimice la fricción para el cliente (que usa el sistema ocasionalmente)
3. Sea confiable para el staff (que lo usa diariamente)
4. No requiera que el despacho gestione una base de contraseñas
5. Sea implementable con Supabase Auth

---

## Alternativas evaluadas

### Contraseñas tradicionales
**Ventajas:** Los usuarios lo conocen. Funciona offline.
**Desventajas:**
- Los usuarios reutilizan contraseñas — una brecha en otro servicio compromete FiscalOS
- El despacho se vuelve responsable de una base de contraseñas hasheadas
- Requiere flujo de recuperación de contraseña (que en la práctica es un magic link)
- El 40% de los tickets de soporte en aplicaciones con contraseñas son recuperaciones
- Para usuarios ocasionales (los clientes), olvidar la contraseña es la norma

### OAuth (Google / Microsoft)
**Ventajas:** Sin contraseñas, UX familiar, los usuarios ya confían en Google/Microsoft.
**Desventajas:**
- Los clientes del despacho no necesariamente tienen Google Workspace o Microsoft 365
- Si el proveedor OAuth tiene un incidente, los usuarios de FiscalOS no pueden entrar
- Requiere gestionar la revocación de tokens OAuth
- Añade dependencia de terceros en el flujo crítico de autenticación
- El consentimiento de OAuth puede confundir a usuarios no técnicos ("FiscalOS quiere acceder a tu cuenta de Google")

### Passkeys (WebAuthn)
**Ventajas:** Estándar emergente, más seguro que contraseñas, sin servidor de contraseñas.
**Desventajas:**
- El soporte del ecosistema aún no es universal (especialmente en dispositivos más antiguos)
- La experiencia de recuperación cuando se pierde el dispositivo es compleja
- Supabase Auth no tiene soporte completo de Passkeys en el MVP
- Prematuro para el perfil de usuario del MVP

### SMS OTP
**Ventajas:** Los usuarios están familiarizados con los códigos de 6 dígitos.
**Desventajas:**
- Requiere proveedor de SMS (Twilio, etc.) con costo por mensaje
- El número de teléfono se vuelve un segundo dato sensible a gestionar
- Los ataques de SIM swapping comprometen la autenticación por SMS
- Muchos clientes del despacho prefieren el email como canal formal

### Magic Links (OTP por email)
**Ventajas:**
- Sin contraseñas — no hay base de contraseñas que proteger ni fugas que gestionar
- El email ya es el canal de comunicación formal del despacho con sus clientes
- Un solo click — la menor fricción posible para usuarios ocasionales
- El flujo de recuperación es el mismo flujo de login — no hay flujos separados
- Supabase Auth lo implementa de forma robusta y probada
- Los tokens son de un solo uso (OTP) y tienen expiración — no son reutilizables
- El vector de ataque (email comprometido) ya existiría con cualquier método basado en email

**Desventajas:**
- Requiere acceso al email para cada login (sin recuerdo de dispositivo largo)
- Si el servidor de email falla, los usuarios no pueden entrar
- En dispositivos donde el email no está configurado, la experiencia es torpe
- La sesión expira — el usuario tendrá que solicitar un nuevo magic link periódicamente

---

## Decisión

**Magic Links** via Supabase Auth es el método de autenticación principal para todos los roles en el MVP.

La decisión se basa en que el mayor riesgo de seguridad en una aplicación B2B con datos sensibles no es el sofisticado atacante — es el usuario que reutiliza contraseñas de servicios comprometidos. Los magic links eliminan ese vector por completo.

Para el perfil de usuario del MVP:
- El cliente del despacho prefiere la simplicidad: un email con un link, sin recordar contraseñas
- El staff del despacho usa el sistema diariamente — la sesión de 1 hora con refresh automático minimiza la necesidad de re-autenticar

---

## Consecuencias

### Positivas
- Eliminación de la clase completa de vulnerabilidades relacionadas con contraseñas
- No se requiere gestionar recuperación de contraseñas
- El flujo de onboarding es idéntico al flujo de login — un solo mecanismo
- Los tokens OTP expiran y son de un solo uso — window de ataque mínima

### Negativas
- Los usuarios deben tener acceso al email para iniciar sesión
- Dependencia del servicio de email de Supabase para la entrega de magic links
- Si el email del usuario cambia, el acceso debe ser restablecido por un admin

### Mitigaciones
- La sesión se mantiene viva con refresh automático — los usuarios del staff no necesitan autenticarse varias veces al día
- Para producción: configurar un dominio de email personalizado para mejorar la entrega y evitar filtros de spam
- Monitorear la tasa de entrega de magic links como métrica de confiabilidad del sistema de auth

### Decisiones derivadas
- La expiración del magic link es de 1 hora — suficiente para que el usuario abra su email sin ser demasiado permisiva
- La sesión del usuario expira en 1 hora pero se renueva automáticamente si el usuario está activo
- El refresh token tiene una expiración de 30 días — si el usuario no entra en 30 días, deberá solicitar un nuevo magic link
- En una versión futura (v2), se evaluará agregar Passkeys como método alternativo para usuarios del staff
