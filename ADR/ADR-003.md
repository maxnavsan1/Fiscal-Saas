# ADR-003 — Por qué Supabase como plataforma

- **Estado:** Aceptado
- **Fecha:** 2026-06-30
- **Decisores:** Arquitectura inicial del proyecto

---

## Contexto

FiscalOS requiere múltiples servicios de infraestructura para funcionar: autenticación de usuarios, base de datos relacional, almacenamiento de archivos y comunicación en tiempo real. Estos servicios pueden construirse de forma independiente con distintas herramientas, o pueden obtenerse como una plataforma integrada.

En la etapa inicial de un producto, la velocidad de desarrollo es un recurso más escaso que el control de infraestructura. La pregunta es: ¿cuánta infraestructura debemos construir nosotros mismos?

---

## Problema

Necesitamos una plataforma que provea:
1. **Auth** — Magic links, JWT, gestión de sesiones
2. **Base de datos** — PostgreSQL con RLS
3. **Storage** — Almacenamiento de archivos con presigned URLs
4. **Realtime** — Notificaciones push sin polling

Construir cada uno de estos componentes de forma independiente es posible pero costoso en tiempo. El riesgo de infraestructura casera es real: la autenticación segura es difícil de implementar correctamente; el storage de archivos con presigned URLs y control de acceso requiere integración con S3 o similar.

---

## Alternativas evaluadas

### Firebase (Google)
**Ventajas:** Muy maduro, excelente DX, Firestore tiene realtime nativo, amplia adopción.
**Desventajas:** Firestore es NoSQL — incompatible con el modelo relacional requerido y con RLS. Las reglas de seguridad de Firestore son expresivas pero no equivalentes a RLS en PostgreSQL. No hay PostgreSQL. El lock-in con Google Cloud es significativo. El pricing puede escalar rápido.

### AWS Amplify
**Ventajas:** Integración con todo el ecosistema AWS, alta disponibilidad, escala ilimitada.
**Desventajas:** Curva de configuración muy alta. Amplify abstrae demasiado sobre servicios AWS, con configuraciones que pueden volverse opacas. Para un equipo pequeño, la complejidad operativa es excesiva en el MVP.

### Stack custom: Postgres propio + Auth0/Clerk + AWS S3 + Pusher/Ably
**Ventajas:** Control total, no hay vendor lock-in en la plataforma.
**Desventajas:** Múltiples vendors, múltiples integraciones, múltiples facturas, múltiples SDKs. La complejidad de integrar 4-5 servicios correctamente con RLS sincronizado es considerable. Los costos sumados suelen superar a Supabase.

### Supabase
**Ventajas:**
- PostgreSQL gestionado con RLS — nuestra decisión de seguridad central
- Auth con magic links, OAuth, MFA incluido
- Storage con buckets privados, presigned URLs y RLS integrado con la base de datos
- Realtime basado en Postgres logical replication — sin servicio adicional
- CLI local para desarrollo — toda la plataforma corre localmente en Docker
- Open source — si Supabase desaparece, los datos siguen siendo PostgreSQL y podemos auto-hostear
- SDK TypeScript tipado que genera tipos desde el esquema
- Pricing predecible, con tier gratuito generoso para desarrollo

**Desventajas:**
- Vendor lock-in parcial (el SQL es portable, pero las políticas de RLS y Storage requieren adaptación)
- La plataforma es relativamente joven — algunos servicios en maduración
- El soporte de Supabase puede ser lento en plan gratuito
- El Realtime tiene limitaciones de escalabilidad en planes básicos

---

## Decisión

**Supabase** es la plataforma de infraestructura base del MVP.

La decisión se basa en tres factores determinantes:
1. PostgreSQL con RLS es un requisito no negociable — Supabase lo provee de forma gestionada
2. La integración nativa entre Auth, RLS y Storage elimina la complejidad de sincronizar permisos entre múltiples servicios
3. El CLI local de Supabase permite replicar el ambiente de producción exactamente en desarrollo — sin mocks, sin sorpresas

El riesgo de lock-in es real pero aceptable: el núcleo del sistema (PostgreSQL + esquema + RLS) es completamente portable. Si fuera necesario migrar, el trabajo sería re-implementar Storage y Auth, no reescribir la base de datos.

---

## Consecuencias

### Positivas
- Un solo SDK para Auth + DB + Storage + Realtime
- Los tipos de TypeScript se generan automáticamente desde el esquema de PostgreSQL con `supabase gen types`
- Las políticas de RLS se aplican a Storage también — un documento en el bucket respeta los mismos permisos que su registro en la base de datos
- El CLI local garantiza paridad entre development y production
- No necesitamos gestionar certificados SSL, backups diarios ni replicación — Supabase lo hace

### Negativas
- Las políticas de RLS requieren conocimiento de Postgres SQL — no se escriben en el lenguaje de la aplicación
- El Storage de Supabase no incluye backups automáticos en el plan base — se necesita replicación a S3 externo
- Si Supabase tiene un incidente, Auth, DB y Storage caen juntos (single point of failure de plataforma)
- Las migraciones de esquema se gestionan en paralelo desde Supabase CLI (frontend) y Alembic (FastAPI) — requiere coordinación

### Decisiones derivadas
- El proyecto Supabase de desarrollo se corre localmente con `supabase start` — nunca se apunta al proyecto de production desde el entorno local
- Se mantienen proyectos Supabase separados para staging y production
- Los backups de Storage se replican periódicamente a un bucket externo como medida adicional de seguridad
- La dependencia en Supabase se encapsula en `src/lib/supabase/` — el resto del código no importa directamente de `@supabase/supabase-js`
